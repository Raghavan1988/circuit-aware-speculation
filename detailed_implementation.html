◆ First, the big idea

Big AI models are slow because they write one word at a time. Each word needs a full pass through the model.

Speculative decoding is a trick to go faster. It uses two models:

  • A small, fast model (the "draft") guesses the next few words.
  • A big, accurate model (the "target") checks all those guesses at once, in a single pass.

Right guesses are kept for free — several words for the price of one big check. Wrong guesses are thrown away. The key promise: the final text is exactly what the big model would have written alone. We never trade quality for speed.

Two words we use a lot:

  • Accept — the big model agrees with a guess, so we keep it.
  • Draft length (L) — how many words the small model guesses before we check. Too few wastes the big model's parallel check; too many wastes the small model's time on words that get thrown out.

A The engine (the machine that does the work)

Our engine is exact and lossless: the words it produces are identical to what the big model would produce alone. We treat the big model's top pick as the correct answer, always. We keep two jobs separate on purpose (rule D021): a hookable path that's easy to inspect (used for all science), and a fast path used only for timing (it never touches the science path).

The parts of the engine

PartWhat it does
Target (big, 7B)The judge. One pass per round; checks all the guesses. Its top pick is the "correct" answer. Qwen2.5-7B-Instruct, fixed version, bf16.
Draft (small, 0.5B)The guesser. Proposes up to L words, one at a time. Shares the big model's vocabulary.
Verify + commitThe referee. Keeps the longest run of guesses that match the big model, then adds one guaranteed-correct word. Plain code, tested on a normal CPU.
Memory caches (KV)Both models remember what they've read, so they don't redo work. After each round, memory is trimmed to exactly what was accepted.
Signal recorderNotes how unsure the small model was for every guess (its "entropy" and "margin"). This is what the smart controller reads later.
ControllerThe decision-maker. Before each round it picks L from the menu {0,1,2,3,4,6,8}. (0 means "skip guessing this round.")

One round, step by step

  1. Pick a length. The controller chooses L.
  2. Guess. The small model proposes up to L words and records its confidence.
  3. Check. The big model does one pass over all the guesses.
  4. Commit. Keep the matching guesses plus one correct word.
  5. Trim memory. Roll both caches back to what was accepted.
  6. Stop? Halt at the end-of-text signal or the length limit.
  7. Record. Save a fresh confidence signal for the next round.
  8. Log. Write everything down and repeat.
The clever trick that saves us money

Every round we record whether each guess matched, at every position — even past where guessing stopped. We call this the match vector. With that full record, we can ask "what would a different draft length have done?" without re-running any model. One expensive run (guessing 8 words each time) scores every possible length setting for free, just by reading the saved file. So all our later analysis is replaying saved data, not paying for new GPU runs.

How we know it's correct

A test checks the engine's output word-for-word against the big model alone — it must match every time. In full-precision math the two agree to about 1 part in 10 million. On the Llama model pair, this check passed 118 out of 118 times. The fast (timing-only) path must re-pass this same check before any of its numbers count.

Three ground rules

  • How we split the data. We divide prompts into a "practice" set and a "final exam" set by whole prompt, before any tuning, then freeze the split. We never split in the middle of one prompt — that would let answers leak from practice into the exam.
  • Where each row came from. Every item carries its license and a source ID. For copyright-sensitive sources, we ship only the row ID plus our own model's output — not the original text.
  • Versioning. Version 2 lives in its own folder, so all the older version-1 results stay valid and untouched.

C How we answer each question

All three questions are answered offline — by replaying the sealed data. Thanks to the match-vector trick, one saved run per prompt tells us the answer for every draft length, every small model, and the acceptance atlas. No new GPU runs needed.

The score we use (and how sure we are)

We measure efficiency: useful words produced, divided by the work spent. We price a small-model guess at one-tenth the cost of a big-model check (a realistic serving assumption).

efficiency = (total words kept) / (total cost)
   cost of a round with length L = 1 + 0.1 × L
   words kept = matching guesses + 1 free correct word

To know how confident we are, we use a prompt-grouped bootstrap: resample whole prompts 2,000 times, report the middle 95% range, plus the chance the true gain is zero or negative.
